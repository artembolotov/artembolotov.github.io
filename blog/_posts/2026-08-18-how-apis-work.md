Это двенадцатая статья из серии [«Внутри интернета»](/blog/internet/). В [прошлый раз](/blog/2026/08/11/how-mobile-networks-work/) мы разбирались с мобильной связью — последним беспроводным кусочком интернета и тем, как телефон ухитряется не разрывать соединение, переходя между сотами. А в самом конце я обещал зайти со стороны, с которой мы в реальной жизни всё чаще и пользуемся интернетом с телефона: со стороны приложений.

Сегодня — про API. Что это вообще такое, какие команды на нём бывают, в каком виде приходят ответы и почему один и тот же сервер одинаково удобно отдаёт данные сайту, мобильному приложению и виджету на часах.

## Приложение — это не сайт

Главное, что стоит понять про приложения, — они устроены не так, как сайты.

Когда вы открываете в браузере сайт банка, происходит вот что: браузер просит у сервера HTML-страницу, получает её, плюс к ней пачку CSS и картинок, и отрисовывает всё это на экране. Сервер прислал готовый макет: где стоит кнопка «Перевести», какого она цвета, какой шрифт у заголовка. Браузер ничего не придумывает — он просто рисует то, что ему сказали.

С приложением картина другая. Когда вы открываете приложение того же банка и видите свой баланс, на экран не приехала никакая HTML-страница. Приложение спросило у сервера ровно одно: «какой остаток на счёте у такого-то клиента?» Сервер ответил: «12 345,67 рублей». А как именно показать эту цифру на экране — какого она будет цвета, каким шрифтом, в какой части экрана, с какой анимацией — приложение уже знает само. Разметка и логика отображения зашиты внутрь, на сервер за ними ходить не надо.

Это и есть принципиальная разница. Сайт получает с сервера готовый внешний вид. Приложение получает с сервера голые данные и собирает внешний вид из них само.

Чтобы такой обмен работал, должен быть общий язык — на котором приложение умеет спрашивать, а сервер умеет отвечать. Этот язык и называется **API**.

## API: окошко вместо обеденного зала

Расшифровывается как Application Programming Interface — «программный интерфейс приложений». На пять минут запомнить полезно, потом можно забыть. Важнее, что это по сути.

API — это набор команд, которые сервер согласен выполнять, и формат, в котором он принимает запросы и возвращает данные. Грубо говоря, это «меню сервера»: вот список того, что он умеет, вот как у него это надо просить, вот как он будет отвечать.

Аналогия — кафе с двумя режимами работы. Если вы садитесь за столик, это сайт. Официант принесёт меню, вы сделаете заказ, на кухне приготовят, и перед вами окажется полностью готовое блюдо: с гарниром, с зеленью, на красивой тарелке, в каком виде кухня сама решит подать.

Если вы подходите к окошку выдачи на вынос, это API. Через окошко вы получаете тот же продукт, но в нейтральном виде: котлета, пюре, соус — каждое в своей коробочке. А как вы это разложите по тарелке у себя дома, посыпете ли укропом и подадите ли с салатом — ваше дело. Кухня сделала свою часть: приготовила еду. Сервировкой занимаетесь вы.

Это самое окошко в терминах API называется **endpoint**, «конечная точка». У каждого API их обычно много: одно для запроса баланса, второе для перевода денег, третье для списка операций. У каждого свой адрес, и приложение знает, в какое окошко за чем идти:

```
GET  https://api.bank.ru/v1/balance
POST https://api.bank.ru/v1/transfer
GET  https://api.bank.ru/v1/operations
```

Приложение знает, куда стучаться. Сервер знает, как обрабатывать запросы по каждому адресу. И обе стороны разговаривают на одном и том же языке — HTTP, том самом, о котором мы [говорили в четвёртой статье](/blog/2026/06/23/http-and-https/).

## Те же методы, что и у браузера

Помните, в четвёртой статье мы разбирали методы HTTP — GET, POST, PUT, DELETE? Тогда они звучали скорее как технические подробности, мимо которых обычный пользователь проходит, не глядя. А для API это центральная штука.

Четыре метода буквально соответствуют четырём естественным действиям над данными:

- **GET** — «покажи мне». Запросить что-то, ничего не меняя. Получить баланс, получить список операций, получить профиль пользователя.
- **POST** — «создай». Отправить серверу новые данные, чтобы он завёл новую запись. Сделать перевод, отправить сообщение, опубликовать пост.
- **PUT** (или близкий к нему **PATCH**) — «измени». Обновить существующую запись. Поменять адрес доставки, отредактировать имя в профиле.
- **DELETE** — «удали». Тут всё понятно.

В жизни это выглядит так. Когда вы в приложении Яндекс.Карт открываете карточку кафе, приложение в фоне делает GET-запрос: «дай данные про вот это конкретное заведение». Когда вы ставите оценку — POST: «вот моя новая оценка». Когда редактируете её — PUT. Когда удаляете — DELETE.

Никаких HTML-страниц в этих запросах не пересылается. Туда-сюда летят только данные: координаты, идентификаторы, рейтинги, тексты. А приложение уже само рисует у себя на экране карточку, кнопки и всё остальное по этим данным.

## JSON: стандартный бланк

Раз приложение и сервер обмениваются голыми данными, эти данные надо в чём-то записывать. Нужен формат, который и сервер на каком-нибудь Python, и приложение на Swift или Kotlin понимали бы одинаково.

Этот формат — **JSON** (JavaScript Object Notation). Его придумали в начале нулевых, и за пятнадцать лет он стал фактически универсальным языком API. Выглядит он вот так:

```json
{
  "balance": 12345.67,
  "currency": "RUB",
  "updated_at": "2026-06-18T10:30:00Z",
  "owner": {
    "name": "Иван Иванов",
    "client_id": 9837245
  }
}
```

Идея простая: пары «ключ — значение», вложенные одни в другие. Значения могут быть числами, строками, массивами, другими объектами. Каждый язык программирования умеет такие данные разбирать и собирать одинаковым штатным образом.

До JSON у API была другая мода — XML, формат, в котором, например, до сих пор делают офисные документы. XML был многословнее, обрабатывался дольше, читался хуже. К середине нулевых JSON начал вытеснять его именно за компактность и простоту, и сейчас почти все новые API — джейсонные.

JSON хорош ещё тем, что человек его тоже легко читает. Если в DevTools посмотреть на ответ от API какого-нибудь современного сайта, общий смысл обычно понятен без подготовки: вот баланс, вот валюта, вот владелец счёта.

## Один сервер — много клиентов

Самое интересное в API начинается, когда задумаешься, кто к нему может подключаться.

Сервер сам по себе ничего не знает про того, кто к нему пришёл. И ему всё равно: веб-версия сайта, iOS-приложение, Android-приложение, виджет на умных часах, плагин для голосового ассистента или сторонний сервис, автоматически собирающий данные. Все они стучатся в одни и те же endpoints, посылают одни и те же запросы и получают одни и те же JSON-ответы. Дальше каждый клиент сам решает, как эти данные показать.

Именно поэтому iOS- и Android-приложения одного и того же сервиса так похожи внешне. Они получают совершенно одинаковые данные из одного и того же API, просто рисуют их разными нативными элементами своей платформы. Дизайнер делает один интерфейс, программисты на двух платформах его реализуют под себя.

И именно поэтому, когда у крупного сервиса падает «бэкенд» — та самая серверная часть с API, — отваливается всё одновременно: и сайт, и оба приложения, и часы, и встроенные интеграции. Эту ситуацию я регулярно вижу и у себя в работе. RedCalendar — это iOS-приложение и сервер, к которому оно ходит за данными. Когда серверная часть по какой-то причине недоступна, приложение у меня в руках ровно так же беспомощно, как и веб-версия любого сервиса в браузере в подобной ситуации: стучит в окошко, никто не отвечает, и кроме крутящегося индикатора показать пользователю особо нечего.

Зато архитектурно эта схема очень удобна. Серверную логику меняешь в одном месте — и сайт, и iOS, и Android начинают показывать новое автоматически, без необходимости выпускать обновление приложения и ждать, пока его установят пользователи.

## Кого пускают в окошко

Не любой API открыт для всех желающих. Сервис погоды может разрешать кому угодно дёргать его за свободные данные. А вот сервер банка обязан выдавать баланс только конкретному клиенту и только после того, как убедится, что это действительно он.

Поэтому большинство API требуют какого-то подтверждения «я имею право здесь спрашивать». Простейшая форма — **API-ключ**: длинная строка из букв и цифр, которую сервер выдаёт клиенту заранее. Клиент прикладывает этот ключ к каждому запросу, сервер сверяется и пускает.

Аналогия — карточка лояльности. Приходите в окошко: «здравствуйте, я клиент номер такой-то, вот моя карточка». Кассир смотрит, видит — да, действительно наш клиент, обслуживает. Без карточки в окошко просто не пускают.

Зачем это нужно. Во-первых, контроль доступа: некоторые операции разрешены только определённым клиентам. Во-вторых, тарификация: платные API считают количество ваших запросов и выставляют по нему счёт. В-третьих, защита от перегрузки: у каждого ключа есть лимит запросов в секунду, и если кто-то ломится в API в десять рук, его ключ просто перестанут пускать до завтра.

Для приложений с пользователями поверх API-ключей надстраивается ещё один уровень. Когда вы открываете приложение банка и вводите код, в обмен на правильный код сервер выдаёт приложению временный токен. Этот токен приложение прикладывает к каждому следующему запросу: «вот мой пропуск, я уже представился, я Иван Иванов». Через несколько часов или дней токен истечёт, и придётся снова подтвердить личность.

Практическое следствие: API-ключи и токены — это, по сути, пароли. С ними нельзя обращаться легкомысленно. Если вы написали небольшую программку, которая ходит в чей-то API с вашим ключом, ни в коем случае не выкладывайте её исходники с этим ключом в открытый репозиторий. Через час до ключа доберутся боты, и владелец API увидит у себя миллион странных запросов от вашего имени.

## Попробуй сам

Главное достоинство публичных API в том, что в большинство из них можно прямо так зайти в браузере и посмотреть глазами. Без приложения, без программы, вообще без ничего.

**Задание первое. Откройте API погоды.**

В адресной строке браузера наберите этот адрес:

```
https://api.open-meteo.com/v1/forecast?latitude=57.77&longitude=40.92&current_weather=true
```

И нажмите Enter. Координаты в адресе — это Кострома. Открыть можно в любом браузере: сайт open-meteo бесплатный и без регистрации.

Браузер покажет вам не страницу, а голый JSON-ответ, что-то в духе:

```json
{
  "latitude": 57.78,
  "longitude": 40.93,
  "current_weather": {
    "temperature": 18.4,
    "windspeed": 3.6,
    "weathercode": 3,
    "time": "2026-06-18T11:00"
  }
}
```

Это ровно тот же ответ, который получило бы приложение погоды у вас на телефоне. Приложение поверх этих цифр нарисует красивую плитку с градусом и солнышком, а браузер показывает данные как есть.

Поменяйте в адресе координаты на свой город — найти их можно за минуту на Яндекс.Картах (правая кнопка по точке — «Что здесь?»). Перезагрузите страницу: JSON обновится, цифры станут вашими.

**Задание второе. Поймайте API на сайте.**

Откройте любой современный сайт — главную ВКонтакте, Озона, Авито или Кинопоиска. Откройте DevTools (`F12` или `Cmd+Option+I`), вкладку **Network**. Сверху найдите фильтр и переключите его на **Fetch/XHR** — это именно API-запросы, без HTML-страницы и картинок.

Теперь поскролльте, кликните на товар, разверните комментарии. В Network начнут появляться новые запросы: это сайт в фоне ходит в собственный API за свежими данными. Кликните на любой — справа во вкладке **Response** увидите тот самый JSON.

То есть современный сайт, по сути, наполовину устроен как приложение. Первоначальная страница приходит как HTML, но дальше всё динамическое подгружается через API в формате JSON, а уже браузерный JavaScript решает, что и куда дорисовать.

## Что дальше

У нас собралась довольно полная картина: язык (HTTP), адреса (IP и DNS), провода и вышки, серверы и хостинги, почта, мобильная связь, теперь API. Всё это разные слои одного и того же интернета, и каждый слой ведёт себя предсказуемо.

Возникает естественный вопрос. А кто это всё придумал? Кто решил, что метод называется GET, а не FETCH? Кто согласовал, что MX-запись в DNS возвращает почтовый сервер? Кто говорит, что JSON — это вот такой формат, а не какой-нибудь другой?

Оказывается, никто конкретно. Интернет — система с распределённым управлением: правила устанавливают несколько независимых организаций, и любой желающий имеет право предложить свою поправку. В следующей статье разберёмся, как устроен этот процесс и почему, например, существует официальный документ про передачу IP-пакетов почтовыми голубями.
